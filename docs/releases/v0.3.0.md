# v0.3.0 — 2026-03-10

v0.3.0 transforms Activity from an audit log query service into a full
platform for understanding what happens inside your control plane. You can now turn
audit logs and Kubernetes events into plain-language summaries, explore them
through a purpose-built UI, stream them in real time, and query them from AI
assistants — all from the same API surface.

This release includes dozens of improvements since v0.2.0.

---

## Human-readable activity feed

The biggest addition in this release: audit logs and Kubernetes events can now
be automatically translated into plain-language summaries like "Alice created
HTTPProxy for myservice.com." You define the translation rules yourself using
ActivityPolicy resources and CEL expressions, so the summaries match the
vocabulary your team already uses.

- Write one ActivityPolicy per resource kind to control how audit events and
  Kubernetes events are turned into Activity records.
- Use CEL expressions for matching (e.g., `audit.verb == 'create'`) and Go
  template syntax for the summary text.
- Test policies safely before deploying with the PolicyPreview API — pass in a
  sample audit log and see the rendered output without affecting live data.
- Query Activity records through the `activity.miloapis.com/v1alpha1` API,
  filterable by actor, resource kind, time range, and more.
- Use ActivityFacetQuery to get distinct values for any field, making it easy
  to build autocomplete and filter UIs.

See [Writing activity policies](../guides/activity-policies.md) for a
step-by-step authoring guide, or [Activity pipeline architecture](../architecture/activity-pipeline.md)
for how translation works under the hood.

## Kubernetes events as a first-class data type

Kubernetes events — the "FailedScheduling" and "BackOff" messages you see with
`kubectl describe` — are now stored, searchable, and streamable alongside audit
logs. This means control plane health signals and user-driven changes live in the same
query interface.

- Search events by resource name, namespace, reason, message, or time range
  using EventQuery.
- Use EventFacetQuery to enumerate distinct reasons or message patterns across
  your event history.
- ActivityPolicy rules can match on Kubernetes events as well as audit logs,
  so you can generate activity summaries from pod restarts, scheduler decisions,
  or any other control plane signal.

See [Event pipeline architecture](../architecture/event-pipeline.md) for how
events are collected and routed.

## Reprocess history when policies change

When you add a new ActivityPolicy or update an existing one, historical audit
logs and events don't automatically get reprocessed. ReindexJob closes that gap:
it backfills Activity records from your stored history so your activity feed
reflects the complete picture, not just events that arrived after the policy was
written.

- Create a ReindexJob targeting a specific policy or time range to trigger
  backfill processing.
- Monitor progress through the job's status conditions — the job reports how
  many records were processed and whether it completed successfully.
- Combine ReindexJob with policy iteration: draft a policy, preview it, deploy
  it, then reindex to apply it retroactively.

See [ReindexJob guide](../guides/reindex-jobs.md) for usage and examples.

## A UI for exploring activity

Activity ships a React component library — `@datum-cloud/activity-ui` on npm —
that you can embed in your own platform UI. Components are designed to work
together as a cohesive activity exploration experience, not just isolated
widgets.

- **Activity feed**: Paginated, filterable list of activity records with
  deep-linkable filter state, so users can share a specific view via URL.
- **Policy editor**: Full-featured editor for ActivityPolicy resources with CEL
  autocomplete, syntax highlighting, and inline validation.
- **PolicyPreview panel**: Test a policy against a real or synthetic audit log
  sample without leaving the editor.
- **ReindexJob management**: Create and monitor reindex jobs directly from the UI.
- **Event explorer**: Search and filter Kubernetes events with the same filter
  primitives as the activity feed.

See [Activity UI guide](../guides/activity-ui.md) for installation and
integration instructions.

## Ask your AI assistant what happened

Activity now includes an MCP (Model Context Protocol) server, which exposes
your activity data to AI tools like Claude. Instead of writing queries manually,
you can ask your AI assistant things like "what changed in the networking
namespace last week?" and get structured answers drawn from real activity records.

- The MCP server exposes tools for querying activities, audit logs, and events
  using natural-language parameters that the AI translates into API calls.
- Works with any MCP-compatible AI client.
- Scoped to the same tenant and permission model as the rest of the Activity API
  — the AI can only see what the configured identity is authorized to see.

See [MCP server guide](../guides/mcp-server.md) for setup and configuration.

### Claude Code plugin

The new `milo-activity` Claude Code plugin builds on the MCP server to provide
guided workflows for common tasks. Instead of knowing which MCP tools to call
and in what order, skills and agents encode multi-step investigation and
authoring patterns so Claude handles the orchestration for you.

- **Incident investigation**: Describe what went wrong and the plugin
  systematically queries recent activity, finds failures, compares against
  baselines, and produces a structured report with a root cause hypothesis.
- **User auditing**: Ask what a specific user did and get a scoped report
  covering resources touched, actions by type, and any failed operations.
- **Policy authoring**: An interactive agent walks you through discovering
  available data, drafting an ActivityPolicy with CEL rules, previewing it
  against real data, and iterating until the output is right.
- **Quick status**: The `/milo-activity:status` command gives a one-shot
  summary of recent control plane activity — useful for standups and handoffs.

Install the plugin from the Datum Cloud marketplace:

```
/plugin marketplace add datum-cloud/claude-code-plugins
/plugin install milo-activity@datum-claude-code-plugins
```

## Real-time streaming

Activity and Event resources now support the Kubernetes Watch API, which means
dashboards and tools can receive instant updates without polling. Changes appear
as soon as they are processed.

- Watch the Activity resource stream to update a live feed as new activities
  are generated.
- Watch EventQuery results to surface control plane health signals in real time.
- Watch APIs follow standard Kubernetes semantics: use `resourceVersion` to
  resume a watch after a disconnect without replaying the full history.

See [Live streaming architecture](../architecture/live-streaming.md) for how
Watch is implemented and when to use it.

## Operational improvements

This release adds several capabilities that make Activity easier to run and
observe in production.

- **Structured logging**: All components now emit structured JSON logs, making
  it straightforward to route logs to your existing aggregation pipeline and
  write log-based alerts.
- **Grafana dashboards**: Pre-built dashboards ship with the deployment,
  including processing failure monitoring so you know immediately when messages
  fail to process.
- **Data pipeline reconnection alerting**: The system now alerts when the data
  pipeline loses and regains its connection to the event bus, surfacing
  transient reliability issues that were previously silent.
- **TLS and mTLS support**: Internal service connections can now be secured
  with TLS, including mutual TLS for environments that require it.
- **IAM roles for all resources**: Every Activity API resource — including
  ReindexJob — now has corresponding IAM role definitions, so you can apply
  least-privilege access control across the full API surface.

See [Observability guide](../operations/observability.md) for dashboard
installation and alerting configuration.

## Stability and compatibility

- Audit log queries now support faceted search for autocomplete and a NOT
  operator for excluding results from query output.
- Fixed OpenAPI schema generation to produce server-side apply compatible
  schemas, resolving field manager conflicts when managing resources with
  `kubectl apply`.
- Aligned tenant scope type definitions with the Milo platform model, fixing
  scope resolution errors when querying at organization or project scope.
- Optimized the events table sort key to use time as the leading component,
  significantly improving query performance for time-range filtered event
  searches.
